# DevFormat 전략 문서 v2

텍스트 워크플로우 도구군 중심의 제품·SEO·수익화 실행안

## 1. 문서 목적

이 문서는 DevFormat의 내부 실행 문서다. 목표는 아이디어 소개가 아니라 아래 항목을 실제로 결정 가능하게 만드는 것이다.

- 무엇을 만들지
- 무엇을 만들지 말지
- 어떤 순서로 만들지
- 어떤 지표로 성공과 실패를 판단할지
- 언제 광고를 붙일지

본 문서는 2026년 3월 10일 기준 현재 저장소 상태와 시장 난이도를 반영해 작성했다.

## 2. 프로젝트 정의

### 2-1. 한 줄 정의

DevFormat은 개발자가 반복적으로 수행하는 텍스트 변환, 리스트 정리, 인코딩/디코딩, 문자열 포맷 변형을 빠르게 처리하는 웹 유틸리티다.

초점은 범용 도구 백화점이 아니라 개발 워크플로우에 바로 들어가는 작은 도구 클러스터에 있다.

### 2-2. 기본 정보

- 사이트명: DevFormat
- 현재 URL: `https://gurioh.github.io/devformat/`
- 현재 호스팅: GitHub Pages
- 현재 배포 형태: 정적 HTML/CSS/JS
- 현재 대표 기능: Multiline to Single Line Converter

### 2-3. 타깃 사용자

- SQL, JavaScript, Python 등에서 문자열 목록을 자주 다루는 개발자
- 복붙 중심의 빠른 변환 도구를 선호하는 실무자
- 설치형 앱보다 즉시 실행되는 웹 유틸리티를 선호하는 사용자
- 포맷 규칙을 매번 설정하지 않고 프리셋으로 끝내고 싶은 사용자

### 2-4. 핵심 사용 상황

- SQL `IN (...)` 절에 값 목록 붙여 넣기
- 줄바꿈 텍스트를 JS 배열, Python 리스트로 변환하기
- 문자열 목록에 따옴표, 구분자, escape 규칙 적용하기
- Base64, URL encoding 같은 주변 작업 빠르게 처리하기
- 반복되는 텍스트 가공을 브라우저에서 즉시 끝내기

## 3. 현재 상태 진단

### 3-1. 현재 구현 상태

현재 저장소는 단일 도구 중심 MVP 단계다.

- 단일 페이지 기반의 multiline converter 구현 완료
- GitHub Pages 배포 완료
- `robots.txt`, `sitemap.xml`, canonical, GA4 등 기본 SEO 자산 존재
- 검색 색인은 초기 단계
- 다중 도구 운영 구조는 미완성
- 템플릿화된 페이지 생성 체계는 없음

### 3-2. 현재 가치

현재 DevFormat의 실제 가치는 `프리셋이 있는 multiline converter`다.

아직은 `도구 스위트`라고 부르기 어렵다. 따라서 당장 필요한 것은 도구 숫자 확대가 아니라, 품질 있는 도구 페이지를 반복 생성할 수 있는 구조다.

### 3-3. 현재 한계

- 루트 페이지 한 개에 기능과 브랜드가 모두 몰려 있다
- 다중 도구 URL 구조가 없다
- 페이지 템플릿이 없어 새 도구를 추가할 때 메타, 예제, FAQ, 내부 링크가 누락될 가능성이 높다
- 현재 차별화 자산인 프리셋이 제품 전략의 중심 개념으로 정리되어 있지 않다
- 대형 키워드 경쟁을 고려한 범위 제한이 문서 수준에서 명확하지 않다

## 4. 핵심 문제 정의

기존 전략의 핵심 가정은 `도구 수 확대 = SEO 성장 = 광고 수익`이었다. 이 논리는 현재 단계에서 지나치게 단순하다.

문제는 도구 수 자체가 아니라 아래 세 가지다.

- 검색 의도와 잘 맞는 페이지를 반복적으로 만들 수 있는가
- 사용자가 실제 업무에서 다시 찾을 이유가 있는가
- 내부 링크와 프리셋 체계로 제품 군이 하나의 클러스터처럼 보이는가

DevFormat은 초기에 이 세 가지를 풀어야 한다.

## 5. 포지셔닝

### 5-1. 핵심 메시지

`Preset-driven text tools for developers`

설정을 매번 조합하는 대신, 자주 쓰는 결과 포맷을 한 번에 만든다.

### 5-2. 시장 관점

대형 경쟁사는 도구 수가 많지만 경험이 산만하고, 많은 페이지가 비슷한 설명과 광고 배치에 의존한다.

DevFormat은 초기에 `텍스트 변환 워크플로우`라는 좁은 영역에서 밀도 있게 이기는 전략을 취한다. 확장은 이 문제군 안에서 제품 밀도가 확보된 뒤 진행한다.

### 5-3. 차별화 포인트

- 프리셋 중심 UX
- 결과가 예측 가능한 양방향 변환
- 페이지별 예제와 사용 시나리오 제공
- 관련 도구 간 빠른 이동과 내부 링크 구조

### 5-4. 차별화로 보지 않는 요소

아래 항목은 운영 원칙일 수는 있지만 핵심 moat로 간주하지 않는다.

- 다크모드 퍼스트
- 광고 최소화
- 단순히 빠른 로딩이라는 표현

이 요소들은 품질 기준에는 포함되지만 제품 정체성의 중심에는 두지 않는다.

## 6. 제품 전략

이 섹션은 DevFormat의 실제 실행 우선순위를 정하는 기준이다.

### 6-1. 제품 원칙

- 초기에는 한 문제군에 집중한다
- 새 도구는 반드시 기존 도구와 사용 맥락이 연결되어야 한다
- 각 도구는 단순 기능이 아니라 실무 결과 포맷을 제공해야 한다
- 기능 개수보다 페이지 품질과 검색 의도 일치를 우선한다
- 도구 페이지마다 예제, FAQ, 관련 도구 링크를 기본 포함한다

### 6-2. 1차 도구 클러스터

초기 8주 동안 우선 개발할 도구군은 아래와 같다.

1. Multiline to Single Line Converter
2. Quote / Unquote List
3. Delimited Text Split / Join
4. Escape / Unescape String
5. Base64 Encode / Decode
6. URL Encode / Decode
7. Slugify / Case Converter
8. JSON Minify / Prettify

이 8개는 모두 텍스트 워크플로우와 직접 연결되며, 내부 링크 클러스터를 만들기 쉽다.

### 6-3. 후순위 또는 제외 대상

아래 도구는 초기 범위에서 제외하거나 후순위로 둔다.

- Regex Tester
- Diff Checker
- JWT Decoder
- Hash Generator의 확장형 옵션
- Color Converter

이유는 간단하다.

- 경쟁이 강하다
- 제품 포커스가 흐려진다
- 초기 텍스트 워크플로우 클러스터와 연결성이 약하다

### 6-4. 프리셋 전략

프리셋은 UI 옵션이 아니라 DevFormat의 핵심 제품 자산이다.

초기 대표 프리셋은 아래와 같다.

- SQL IN
- JS Array
- Python List
- CSV Row
- Quoted Lines
- Env Var List
- Markdown Bullet
- Newline to Comma
- Pipe-separated
- Tab-separated

프리셋 운영 원칙은 다음과 같다.

- 도구 설명보다 프리셋 가치가 먼저 보이게 한다
- 각 프리셋은 입력 예시, 결과 예시, 사용 상황을 함께 제공한다
- 프리셋 클릭과 복사 이벤트를 측정한다
- 데이터 기반으로 인기 프리셋의 노출 순서를 조정한다

## 7. 정보 구조와 공개 인터페이스

### 7-1. URL 구조

향후 구조는 아래를 기본으로 한다.

- `/` : 도구 카탈로그이자 브랜드 랜딩 페이지
- `/multiline/`
- `/quote/`
- `/split-join/`
- `/escape/`
- `/base64/`
- `/url-encode/`
- `/slugify/`
- `/json/`

현재 루트 페이지가 대표 도구 역할을 하고 있더라도, 1주차 구조 전환에서 `/`는 카탈로그로 이동하는 것을 기본 방침으로 한다.

### 7-2. 경로 정책

- GitHub Pages 프로젝트 사이트 기준 상대경로를 사용한다
- 루트 절대경로(`/json`) 사용을 피한다
- 각 도구는 독립적으로 indexable 해야 한다
- canonical은 최종 공개 URL 기준으로 명시한다

### 7-3. 도구 taxonomy

초기 taxonomy는 아래 네 범주만 사용한다.

- Text Transform
- Encoding
- String Format
- Structured Text

새 도구는 반드시 이 분류 중 하나에 속해야 하며, 별도 범주가 필요하면 후순위 후보로 이동한다.

## 8. SEO 전략

### 8-1. 기본 원칙

- 한 툴은 한 랜딩 페이지를 가진다
- 한 페이지는 하나의 대표 검색 의도를 가진다
- 각 페이지는 제목, 설명, 예제, FAQ, 관련 도구 링크를 포함한다
- 대형 헤드 키워드보다 전환 의도가 높은 롱테일을 우선한다

### 8-2. 타깃 키워드 방향

초기 타깃은 아래처럼 텍스트 변환 작업에 바로 연결되는 롱테일 중심으로 잡는다.

- `multiline to sql in clause`
- `newline to comma separated list`
- `text to javascript array`
- `text to python list`
- `quote each line online`
- `escape string for json`
- `base64 decode string online`
- `url encode query param online`

헤드 키워드는 보조 목표로 둔다. 예를 들어 `json formatter online` 같은 쿼리는 페이지를 만들더라도 초기 핵심 승부처로 삼지 않는다.

### 8-3. 페이지 템플릿 요구사항

모든 도구 페이지는 아래 요소를 공통으로 포함해야 한다.

- H1
- 한 줄 가치 제안
- 즉시 사용 가능한 입력/출력 영역
- 대표 프리셋
- 예제 2개 이상
- 짧은 설명
- FAQ 3개
- 관련 도구 링크
- canonical
- Open Graph 메타
- structured data
- indexable URL

### 8-4. 콘텐츠 원칙

- 도구 소개보다 사용 상황을 먼저 설명한다
- 얇은 설명문 대신 실제 입력/출력 예제를 제공한다
- FAQ는 검색 의도와 직접 연결되는 질문만 넣는다
- 같은 설명을 여러 페이지에 복붙하지 않는다

## 9. 측정 체계와 KPI

초기 단계의 운영 판단은 트래픽 과장치가 아니라 실제 지표로 한다.

### 9-1. 필수 측정 이벤트

- 도구별 페이지 진입
- 프리셋 선택
- 복사 버튼 클릭
- 입력 완료 후 결과 생성
- 관련 도구 링크 클릭

### 9-2. 초기 8주 KPI

- 게시된 indexable tool pages: 8개 이상
- Search Console에 실제로 색인된 페이지: 6개 이상
- 최근 28일 검색 impressions: 200 이상
- 롱테일 키워드에서 첫 organic clicks 발생
- 상위 3개 도구에서 평균 engagement time 40초 이상
- 프리셋 선택 이벤트 누적 100회 이상

위 KPI는 공격적 성장 목표가 아니라, 제품-검색 적합성을 확인하기 위한 최소 기준이다.

### 9-3. 사용하지 않는 지표

초기 단계에서는 아래 지표를 핵심 KPI로 사용하지 않는다.

- 월간 PV 추정표
- 광고 RPM 비교표
- 도구 수 대비 선형 수익 추정

이 수치들은 실행 우선순위를 왜곡할 가능성이 크다.

## 10. 수익화 전략

광고는 1단계 목표가 아니라 2단계 목표다.

### 10-1. 우선순위

1. 페이지 품질
2. 검색 색인과 내부 링크
3. 도구 클러스터 완성도
4. 정책 및 신뢰 페이지
5. 광고 적용

### 10-2. 광고 도입 조건

AdSense 신청 또는 실제 광고 배치는 아래 조건 충족 이후 검토한다.

- 색인된 도구 페이지 최소 8개
- 개인정보처리방침, 이용안내, About 페이지 존재
- 각 페이지가 얇은 콘텐츠가 아닐 것
- 모바일 경험을 해치지 않을 것
- 내부 링크 구조가 완성되어 있을 것

### 10-3. 광고 배치 원칙

- 초기에는 상단 및 하단 배너 중심
- 좌우 sticky 광고는 데이터 확인 후 판단
- 모바일에서는 입력 영역 위 또는 중간 삽입을 피한다
- 광고 수보다 작업 흐름 방해 최소화를 우선한다

## 11. 4주 실행 로드맵

### 11-1. 1주차: 구조 전환

- 도구 페이지를 반복 생성할 수 있는 템플릿 구조 정의
- 공통 메타, 공통 헤더/푸터, 관련 도구 링크 구조 설계
- 툴별 URL 정책 정리
- 현재 단일 페이지를 템플릿 기준으로 분해할 설계안 작성
- 루트 `/`를 카탈로그로 바꾸는 전환 계획 수립

### 11-2. 2주차: 1차 도구 클러스터 확장

- Multiline 관련 3개 도구 추가
- Quote, Join, Split류 프리셋 확장
- 각 페이지에 예제와 설명 추가
- Search Console 제출 갱신

### 11-3. 3주차: 주변 유틸 추가

- Base64 Encode / Decode 추가
- URL Encode / Decode 추가
- Slugify 또는 Case Converter 추가
- 관련 도구 내부 링크 완성

### 11-4. 4주차: 신뢰 및 수익화 준비

- 개인정보처리방침 페이지 추가
- About 페이지 추가
- 도구 카탈로그 페이지 완성
- structured data 반영
- 광고 적용 전 UX 검토

## 12. 리스크와 대응

### 12-1. 주요 리스크

- 대형 키워드 경쟁이 심해 색인이 되어도 클릭이 적을 수 있다
- 도구 수만 늘리면 얇은 페이지 묶음이 될 수 있다
- 광고가 초기 UX를 해칠 수 있다
- GitHub Pages 구조상 라우팅과 절대경로 실수가 발생할 수 있다
- 페이지 수가 늘면서 메타, 링크, 사이트맵 관리가 깨질 수 있다

### 12-2. 대응 원칙

- 롱테일 키워드 중심으로 우선순위를 잡는다
- 공통 페이지 템플릿을 강제한다
- 광고는 조건 충족 이후에만 도입한다
- 상대경로와 프로젝트 사이트 경로 규칙을 고정한다
- 사이트맵 생성과 검수 체크리스트를 운영한다

## 13. 성공 기준

### 13-1. 초기 성공 기준

- 8주 내 8개 이상의 품질 있는 indexable pages 확보
- Search Console 기준 의미 있는 impressions 증가 확인
- 일부 롱테일 키워드에서 첫 클릭 발생
- 상위 3개 도구의 재사용 지표 확인
- 프리셋 사용 데이터로 다음 도구 우선순위 결정 가능

### 13-2. 중장기 성공 기준

- 텍스트 워크플로우 클러스터 안에서 명확한 내부 링크 네트워크 완성
- 광고 도입 전후 UX 하락이 통제 가능
- `DevFormat = text workflow tools`라는 연상이 가능할 정도의 포지셔닝 확보

## 14. 하지 않을 일

초기 단계에서는 아래 항목을 의도적으로 하지 않는다.

- 범용 개발자 도구 포털처럼 확장하기
- 경쟁이 강한 대형 키워드를 정면 승부 목표로 잡기
- 광고 승인만을 위해 얇은 페이지를 양산하기
- 다크모드나 미니멀 디자인을 핵심 전략처럼 포장하기
- 아직 없는 도구를 이미 갖춘 것처럼 문서화하기

## 15. 문서 검토 기준

이 문서는 아래 질문에 모두 답할 수 있어야 한다.

- 현재 제품 상태와 과장 없이 연결되는가
- 무엇을 만들지와 만들지 않을지가 명확한가
- 첫 4주의 실행 순서가 흔들리지 않는가
- 각 도구 추가의 이유를 설명할 수 있는가
- 성장 논리가 도구 수가 아니라 검색 의도와 제품 일관성에 기반하는가
- 수익화가 제품 완성도 이후 단계로 정리되어 있는가

## 16. 기본 가정과 디폴트

- 문서 목적은 내부 실행 문서다
- 문서 언어는 한국어 중심으로 하되 필요한 영문 키워드를 병기한다
- 성장 방향은 텍스트 워크플로우 집중이다
- 숫자 톤은 보수적으로 유지한다
- 초기 단계에서는 도구 수 최대화보다 문제군 밀도를 우선한다
- 광고는 초반 KPI가 아니라 후행 단계로 둔다
- 프리셋은 UI 옵션이 아니라 제품 핵심 자산으로 간주한다

## 17. 다음 실행 항목

이 문서를 기준으로 다음 구현 단계의 기본 순서는 아래와 같다.

1. 현재 단일 페이지를 카탈로그 + 도구 페이지 구조로 분해하는 설계
2. 공통 도구 템플릿 정의
3. Multiline, Quote, Split/Join 도구군 우선 추가
4. 정책 페이지 및 내부 링크 구조 보강
5. KPI 측정 이벤트 반영

이 순서는 별도 반대 근거가 없는 한 기본 실행 순서로 유지한다.
